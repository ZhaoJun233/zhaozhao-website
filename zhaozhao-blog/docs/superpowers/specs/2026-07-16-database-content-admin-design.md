# 全站数据库内容后台设计

日期：2026-07-16

## 目标

将 233昭个人博客从“Decap CMS 修改仓库文件”迁移为“Astro SSR 直接读写 SQLite”的单管理员动态网站。文章、项目、分类、友链和所有页面配置使用同一个数据库与同一套后台管理，不再保留两套内容源。

## 约束

- 后台只有一名管理员。
- 当前通过 Docker 在本机部署，站点端口仅绑定 `127.0.0.1`。
- 后台保存后，前台刷新即可看到修改结果，不执行构建或重启。
- 现有 JSON、Markdown、图片和页面内容必须完整迁移。
- SQLite 数据文件必须通过 Docker volume 持久化。
- 数据库密码和会话密钥只从环境变量读取。

## 架构

应用保留单个 Astro SSR 服务。该服务同时负责读者页面、公开读取 API、管理员页面、管理 API、认证和 SQLite 数据访问。原 Decap CMS、`decap-server` 与独立 `cms` 容器在迁移验证完成后移除。

内容请求路径：

1. 浏览器请求 Astro 页面或公开 API。
2. 运行时内容仓库从 SQLite 读取结构化数据。
3. Markdown 正文通过现有 `marked` 渲染器生成 HTML 和标题索引。
4. 页面使用与当前模板一致的数据结构渲染。

管理请求路径：

1. 管理员在 `/admin/` 输入密码。
2. 服务端校验 `ADMIN_PASSWORD`，成功后签发 HttpOnly、SameSite=Strict 的会话 Cookie。
3. 管理页面通过同源 `/api/admin/*` 读写数据。
4. 每次写操作使用事务、服务端校验和更新时间戳。

## 数据模型

### 系统表

- `schema_migrations`：记录数据库结构版本。
- `admin_sessions`：保存会话摘要、创建时间与过期时间，不保存明文令牌。
- `site_settings`：使用唯一 `key` 和经过 schema 校验的 JSON `value` 保存低频页面配置。

`site_settings` 包含：`profile`、`navigation`、`homepage`、`about`、`guestbook`、`credits`、`page_copy` 与 `artwork`。

### 内容表

- `categories`：名称、slug、说明、排序、启用状态。
- `posts`：slug、标题、摘要、正文、发布时间、更新时间、草稿、分类、标签 JSON、封面、精选、系列和规范链接。
- `projects`：slug、标题、摘要、正文、日期、状态、标签 JSON、封面、仓库链接、演示链接、精选和排序。
- `friends`：名称、网址、介绍、兴趣标签 JSON、排序、启用状态和更新时间。
- `friend_page`：友链页标题、说明、提示、空状态和邀请区域文案。

分类删除前必须确认未被文章引用。文章与项目 slug 唯一。友链、分类和项目排序使用显式整数顺序，并在事务中重排。

## 初始化与迁移

首次启动时执行版本化迁移。如果业务表为空，则读取当前文件并导入：

- `src/data/*.json` 导入设置、分类、友链页和友链。
- `src/content/posts/*.md` 导入文章元数据与正文。
- `src/content/projects/*.md` 导入项目元数据与正文。

导入写入单个事务。任何文件校验失败都会回滚，避免半迁移数据库。导入成功后记录种子版本，后续启动不重复覆盖后台数据。

原内容文件保留为迁移来源和可读备份，但运行时不再读取它们。确认数据库版本稳定后，Decap 配置及服务依赖删除。

## 后台界面

后台采用安静、紧凑的工作台布局，不沿用读者页面的大标题构图。

- 左侧导航：概览、文章、项目、分类、友链、页面内容、媒体与数据。
- 顶部区域：当前模块名称、搜索、保存状态和退出按钮。
- 列表页：搜索、状态筛选、排序、分页、创建按钮和行级编辑/删除操作。
- 编辑页：明确字段分组、必填提示、服务端错误定位、保存与取消。
- 友链：新增、编辑、删除、启停、上移和下移。
- 页面内容：个人资料、导航、首页、关于、留言、鸣谢及通用文案使用结构化表单。
- 数据：下载全量 JSON 备份和导入备份；导入前展示内容统计并要求二次确认。

第一阶段不实现多人角色、审批流、定时发布和对象存储。

## API

公开 API 保持现有路径，并改为数据库读取：

- `GET /api/content`
- `GET /api/posts`
- `GET /api/posts/:slug`
- `GET /api/projects`
- `GET /api/projects/:slug`
- `GET /api/search`

管理 API 使用 `/api/admin` 前缀：

- `POST /api/admin/session`、`DELETE /api/admin/session`
- 文章、项目、分类、友链的列表、详情、创建、修改和删除接口
- `GET`、`PUT /api/admin/settings/:key`
- 排序接口只接收有序 ID 列表，并在事务内验证集合完整性
- `GET /api/admin/export`、`POST /api/admin/import`

API 返回统一 JSON 错误结构：`{ error, fieldErrors? }`。所有修改请求验证 Content-Type、会话和请求体大小。

## 认证与安全

- `ADMIN_PASSWORD` 必填；生产启动缺失时直接报错。
- 密码使用恒定时间比较；会话令牌使用加密安全随机数生成。
- 数据库只保存会话令牌 SHA-256 摘要。
- Cookie 设置 HttpOnly、SameSite=Strict，公开 HTTPS 部署时启用 Secure。
- 管理 API 校验同源请求，并对登录失败进行短时限速。
- Markdown 输出沿用现有渲染路径，不允许管理表单注入可执行脚本。

## 持久化与备份

数据库默认路径为 `BLOG_DATABASE_PATH=/app/storage/blog.sqlite`，Docker 使用命名 volume 挂载 `/app/storage`。SQLite 启用 WAL、外键和 busy timeout。

导出接口生成带 `schemaVersion`、导出时间和各数据集合的 JSON。导入必须先完整校验，再在事务中替换数据。数据库文件级备份可通过复制停止写入后的 volume 快照完成。

## 错误处理

- 数据库不可用时健康检查返回失败，页面返回明确的 503 状态。
- 写入冲突返回 409；字段校验错误返回 422；会话失效返回 401。
- 删除被引用分类返回 409，并附带引用文章数量。
- 管理界面保留未提交表单内容，错误通过页面内状态区和字段提示显示。

## 测试与验收

- 数据库测试：迁移、首次导入、幂等启动、事务回滚、CRUD、排序和关联约束。
- 认证测试：正确/错误密码、会话过期、未登录写入和退出。
- 仓库测试：全部运行时加载函数只读数据库，输出结构兼容现有页面。
- API 测试：状态码、校验、冲突和备份导入导出。
- 浏览器测试：登录、创建草稿、修改分类、增删友链、排序、退出。
- 动态同步验收：容器不重建，后台修改后刷新首页、分类页、文章页和友链页即可看到变化。
- 回归验收：Astro 检查、单元测试、关键路由、桌面与移动端无横向溢出。

## 完成标准

全站所有可维护内容都有数据库记录和后台入口；前台不再依赖 JSON/Markdown 的运行时读取；Decap 服务被移除；SQLite volume 可在容器重建后保留数据；自动迁移、备份、认证和关键管理流程均通过测试。
